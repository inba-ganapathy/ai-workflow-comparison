Critical Findings

Finding #1 — ShipIt Sequential Mode: Evidence Without Code (S2)
ShipIt produced a 5-phase battle-plan, implementation.md, review.md (self-scored 24/25), and validation.md — but wrote zero code. Code inspection confirmed db.py, store/index.ts, and ManagerChat.tsx are unmodified. The validation claims tests pass that were never written. This is a known limitation of running ShipIt without Agent Teams enabled. In sequential mode, the documentation phases complete but the implementation phase cannot execute. ShipIt is designed for multi-agent parallel execution with TeamCreate/SendMessage.
Finding #2 — Innerloop Multi-Agent Overhead on Large Features (S4)
S4 Innerloop consumed 120 tool calls across 7 agents (investigate×4, plan-auditor, A–E). The coordination overhead is well-suited to independent parallel tasks. On focused single-module work (S2, S3), Innerloop was the fastest workflow. The overhead pays off in test coverage (57 tests) and the only complete embedding pipeline.
Finding #3 — Direct Over-Engineering Tendency on Feature Work (S3, S4)
Direct added folders, backlinks, daily notes, and versions to S4 beyond what the prompt required. notesApi.ts references ~8 non-existent endpoints. On S3, Direct added 948 lines vs Innerloop's 536 for equivalent AC coverage. Higher council scores, but 2–6× the lines means more review surface and maintenance burden.
Finding #4 — Only Innerloop Fixed the Highest-Impact Bug (S2)
Bug 7 (approve_plan failure leaves plan stuck in_progress permanently) was the highest-impact remaining bug in S2. Only Innerloop implemented the fix: asyncio.ensure_future inside a done_callback updates plan status to 'failed' on exception. Direct's approach was log-only. ShipIt claimed it was pre-fixed (incorrect — the fix was never applied).

Innerloop — When It Wins

  • Fastest on targeted/focused tasks
  • Bug fix tasks with multiple files
  • When dedicated test files are required
  • When semantic search / embedding needed
  • When the hardest edge cases must be caught

Direct — When It Wins

  • Council score / code review outcomes
  • New feature with explicit AC coverage
  • When codebase pattern adherence is critical
  • When API surface must be preserved exactly
  • Work items with named acceptance criteria

ShipIt — When It Wins

  • Pre-implementation planning and analysis
  • Multi-team work requiring battle plans
  • When Agent Teams ARE available
  • Generating evidence for compliance/audit
  • FTS trigger-based consistency (S4)

Decision Matrix

Use Innerloop when...
Fixing targeted bugs with clear scope where every fix must work. Features decomposable into independent modules. When you need dedicated test files per concern and the hardest edge case matters most (plan-stuck, race conditions). Multi-agent parallelism delivers faster overall wall-time on focused tasks.
Use Direct when...
New feature work where council score and code review outcomes matter. Work items with explicit acceptance criteria requiring named test coverage. When closest adherence to codebase patterns is required. Accept 2–6× the line count in exchange for the highest review scores.
Use ShipIt when...
Agent Teams ARE available. Complex multi-team coordination requiring battle plans, role-based phases, and structured handoffs. Pre-implementation planning and compliance documentation. Do not use for code delivery in sequential mode.
Avoid ShipIt when...
Agent Teams are unavailable. The task requires actual code delivery. You cannot afford false-confidence risk (validated evidence of work that was never done). Any scenario where downstream engineers will assume the code was written because the evidence documents say it was.
