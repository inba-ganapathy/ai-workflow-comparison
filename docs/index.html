AI Workflow Benchmark: Innerloop vs Direct vs ShipIt

Benchmark across 4 engineering scenarios (S1–S4), 3 AI workflows, measuring quality (council score), speed (minutes), and cost (API spend). Single-run measurements per workflow per scenario.

Innerloop

Structured 3-phase: Investigate → Plan → Execute → Review. Dedicated subagents per phase. TDD enforced.

Direct Implementation

No structured workflow: Read files → identify bugs → fix → verify. Single agent, no phases, no review pass.

Ship-It Style

One command. One work item. One PR. Specialist agent pipeline: Analyst → Engineer → Reviewer → Validator. Requires Agent Teams (TeamCreate + SendMessage).

Three Decisive Findings
What the data tells us for strategic workflow selection
01
Direct Implementation is the Default Choice for Well-Specified Tasks
Scoring 20.3/25 on average with the lowest total cost ($16.48) and competitive speed across S1–S3, Direct outperforms InnerLoop on S2, S3, and S4 (scored scenarios). For tasks with clear acceptance criteria, Direct should be the baseline workflow.
02
InnerLoop Shines on Speed-Critical Simple Scenarios
InnerLoop completes S1 in 6.7 min and S2 in 6.8 min — consistently faster than ShipIt and near-instant for low-complexity work. However, its $28.73 total cost and lower council scores on S3/S4 limit its value for complex features.
03
ShipIt's 15.0 Average Score Does Not Justify Its Complexity Overhead
ShipIt scores the lowest on all three evaluated scenarios and is the slowest on S1–S3 (33–41 min). Its only competitive performance is S4 speed (11.1 min). ShipIt should be reserved for well-rehearsed CI/CD scenarios where setup costs are amortized.

What Each Workflow Uniquely Found

InnerLoop Unique findings — S1 Open Task
  • →Plan approval DB race — approve_plan returns success without updating DB when no manager_agent is available; plan stays stuck in "proposed" state permanently
  • →Optimistic UI defeat — ManagerChat adds optimistic message after the API call completes instead of before; defeats the purpose of optimistic UI, adds latency
  • →upsertPlan crash on plan_completed — store.upsertPlan crashes when plan_data is undefined; plan_completed WS event payload carries no plan_data
  • →WS handler payload mismatch — plan_completed / plan_updated handlers pass raw payload as Plan; payload has plan_id not id, missing plan_data
  • →PlanSummary type missing — PlanSummary referenced in api.ts and WorkItemsList.tsx but never declared in types.ts; TypeScript compile error
  • →Thinking status never resets — managerStatus reset to idle in finally block immediately after send; should remain "thinking" until manager response arrives via WS
Source: /tmp/scenario1-innerloop-metrics.json · 6 of 6 bugs verified from primary metrics file
Direct Unique findings — S1 Open Task
  • →Conversations endpoint shape mismatch — GET /api/manager/conversations returns a list but tests/frontend expect a get-or-create single conversation with messages
  • →plan_proposed WS payload not mapped — store/index.ts plan_proposed handler passes raw payload as Plan; plan_data field not extracted from nested plan.steps
  • →useEffect infinite loop — ManagerChat.tsx effect on [messages, plans] calls upsertPlan which mutates plans, re-triggering the effect endlessly
  • →Event loop blocking — _do_commit and _do_open_pr call subprocess.run directly in async context, blocking the entire event loop during git operations
  • →Wrong JSON key for sign-off notes — sign-off prompt returns key "note" (singular) but _sign_off_on_step reads decision.get("notes"); sign-off notes always silently empty
  • →WS events missing handlers — plan_cancelled and plan_started events have no handlers in dispatchWSEvent; plan status never updates in UI until page reload
Source: /tmp/scenario1-direct-metrics.json · 6 of 6 bugs verified from primary metrics file
ShipIt Unique findings — S1 Open Task
  • →Cancel path DB ordering — approve_plan cancel path broadcasts plan_cancelled WS event before updating DB status; UI and DB transiently inconsistent
  • →plan_started handler silent drop — store/index.ts plan_started handler silently skips the status update if the plan isn't already loaded in the store; badge stays "proposed"
  • →updatePlanStep ID mismatch — store/index.ts updatePlanStep raw step-id from WS event never matches DB-prefixed step-id; step progress indicators always stale
  • →create_plan step_type normalisation ambiguity — db.py create_plan step_type normalisation uses a double-lookup dict; ambiguous key resolution when step type appears in multiple categories
  • →Stale steps list in post-exec summary — repo_manager.py _execute_plan reads steps list once before execution; post-execution summary uses stale data if any step was added or modified during run
  • →Session watcher debounce keeps-first — session_watcher.py burst dedup set keeps the first event per file instead of the last; rapid session file changes processed with stale data
Source: /tmp/scenario1-shipit-metrics.json · 6 of 6 bugs verified from primary metrics file
